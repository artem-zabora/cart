# Кошик

Додаток для керування кошиком продуктів.

## Запуск проекта

Для запуску проекта запустіть наступні команди:

1. Установите зависимости:
   ```bash
   composer install
   npm install

2. Перенесіть наявне з .env.example до .env.

3. Підійміть контейнер:
   ```bash
   sail up -d
   
4. Запустіть міграції та заповніть базу данних:
   ```bash
   sail shell
   php artisan migrate --seed

5. Соберите фронтенд:
   ```bash
   npm run dev

6. Також треба буде запустити:
    ```bash
   php artisan key:generate
   php artisan storage:link

# Оцінка стабільності та рекомендації

### Код, який реалізований для роботи з кошиком, має базову функціональність, і в більшості випадків буде працювати стабільно.

### Він включає такі основні можливості:

- Додавання товару до кошика.

- Зміна кількості товарів.

- Видалення окремих товарів із корзини.

- Очищення корзини повністю.

- Розрахунок загальної суми.

- Коли користувач додав товар до кошика, а потім авторизувався чи зареєструвався, його кошик зберігається і далі він вже зв'язаний по user_id замість session_id.

### Звісно, є аспекти, які можна вдосконалити для підвищення стабільності:

- Обробка помилок: У коді є базова обробка помилок, але вона зводиться до показу повідомлень через alert. Це мінімальний рівень обробки. Бажано додати більш інформативні сповіщення та можливість користувачу повторити дію у випадку помилки. Ще було б добре використовувати Exeptions в методах роботи з базою данних.

- Валідація даних: Не перевіряється, чи існує товар у базі даних під час додавання/видалення/збільшення кількості.

- Взаємодія з фронтендом: AJAX-запити обробляються коректно, але взаємодія з інтерфейсом зроблена "на скору руку".

- Безпека: Використання CSRF-токенів забезпечує базовий захист для запитів. Крім того, авторизація та аутентифікація вже реалізовані, але можна додати додаткові перевірки ролей користувачів, щоб визначити рівень доступу до різних функцій.

-  Продуктивність, масштабованість: Код підходить для невеликого проєкту, але в більш складних системах можуть виникнути проблеми з продуктивністю, якщо кількість запитів до сервера буде дуже великою. В реальному проєкті я б обов'язково додав індекси для окремих полів в базі даних. Також використав би зв'язок між таблицями belongsTo з додаванням проміжної таблиці замість hasMany. Ще додав би задачу в Cron для видалення занадто старих записів. Ще, зберігав би кошик не авторизованих користувачів в сесії.

### Додатковий функціонал який можна було б зробити:
- ### Список бажань (Wish List)


#### Опис функціоналу:
 Список бажань дозволить користувачам зберігати товари, які вони планують придбати пізніше. Цей функціонал покращить взаємодію користувача з додатком та збільшить залученість.

#### Як це працює:
Авторизований користувач може додавати товари до списку бажань, і вони будуть збережені в базі даних, пов’язані з його user_id.
Гість також може додавати товари до списку бажань. Дані будуть зберігатися на основі session_id.
Користувач може переглядати список бажань, додавати товари до кошика або видаляти їх зі списку бажань.
